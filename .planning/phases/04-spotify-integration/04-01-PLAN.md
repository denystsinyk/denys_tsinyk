Create `scripts/fetch-spotify-data.js` as a Node.js ESM script. Mirror `scripts/fetch-steam-data.js` structure exactly:

**1. ESM imports and path setup:**
```js
import fs from 'node:fs'
import path from 'node:path'
import { fileURLToPath } from 'node:url'
const __dirname = path.dirname(fileURLToPath(import.meta.url))
const outputPath = path.join(__dirname, '..', 'public', 'data.json')
```

**2. Env var validation (exit(1) on missing — pipeline must fail loudly):**
- `SPOTIFY_CLIENT_ID` — required, exit(1) if missing
- `SPOTIFY_CLIENT_SECRET` — required, exit(1) if missing
- `SPOTIFY_REFRESH_TOKEN` — required, exit(1) if missing

**3. `main()` async function with read-patch-write:**
- Read existing `data.json` via `fs.readFileSync(outputPath, 'utf8')` and JSON.parse
- Inside try block:
  - **Token exchange POST** to `https://accounts.spotify.com/api/token` with:
    - Header: `Content-Type: application/x-www-form-urlencoded`
    - Header: `Authorization: Basic ` + Buffer.from(`${SPOTIFY_CLIENT_ID}:${SPOTIFY_CLIENT_SECRET}`).toString('base64')
    - Body (URLSearchParams): `grant_type=refresh_token`, `refresh_token=SPOTIFY_REFRESH_TOKEN`
  - If `!tokenRes.ok`: throw new Error(`Token exchange HTTP ${tokenRes.status}`)
  - Extract `access_token` from response. ALSO check for `refresh_token` in response — if present and different from stored token, `console.warn('Spotify returned a new refresh token — update SPOTIFY_REFRESH_TOKEN secret manually:', newToken)`. Do NOT auto-update the secret.
  - **Recently-played fetch** GET `https://api.spotify.com/v1/me/player/recently-played?limit=5` with `Authorization: Bearer ${accessToken}`
  - If `!recentRes.ok`: throw new Error(`Recently played HTTP ${recentRes.status}`)
  - Map `recentJson.items` to `SpotifyTrack[]` shape matching `src/types/data.ts`:
    ```js
    {
      name: item.track.name,
      artist: item.track.artists[0]?.name ?? 'Unknown',
      album_art: item.track.album.images[0]?.url ?? null,  // images[0] = largest (640x640)
      spotify_url: item.track.external_urls.spotify,
    }
    ```
  - Set `existing.spotify = { recent_tracks: recentTracks, currently_playing: null }`
  - Set `existing.spotify_ok = true`
  - console.log(`Spotify data fetched: ${recentTracks.length} tracks`)
- In catch block: `console.error('Spotify fetch failed:', err.message)` + `existing.spotify_ok = false` (preserve existing spotify data — do NOT reset it)
- After try/catch: `existing.updated_at = new Date().toISOString()` then `fs.writeFileSync(outputPath, JSON.stringify(existing, null, 2))`
- console.log(`data.json updated at ${outputPath}`)

**Key constraints:**
- `currently_playing: null` always — Phase 4 spec does not include currently-playing music indicator (deferred to v2)
- `images[0]?.url` not `images[2]?.url` — first image is the largest (640×640)
- `artists[0]?.name ?? 'Unknown'` — artists is always an array; use only primary artist
- Read-patch-write only — never replace fields written by other scripts

**Part B — Create RUNBOOK-spotify-token.md:**

Write `.planning/phases/04-spotify-integration/RUNBOOK-spotify-token.md` with these sections:

**Overview:** Why this one-time setup is needed (Spotify Authorization Code Flow requires browser-based user consent for `user-read-recently-played` scope; result is a long-lived refresh_token stored as a GitHub Secret).

**Prerequisites:** Spotify account, access to https://developer.spotify.com/dashboard, access to GitHub repo Settings > Secrets.

**Step 1 — Create Spotify App:**
- Go to https://developer.spotify.com/dashboard → Create app
- App name: anything (e.g. "Portfolio Data")
- Redirect URI: `http://127.0.0.1:3000/callback` (exact match required)
- Check the scope: `user-read-recently-played`
- Save Client ID and Client Secret

**Step 2 — Build the authorization URL:**
```
https://accounts.spotify.com/authorize?client_id=YOUR_CLIENT_ID&response_type=code&redirect_uri=http%3A%2F%2F127.0.0.1%3A3000%2Fcallback&scope=user-read-recently-played&state=xyz
```
Open this URL in a browser while logged into Spotify. Click "Agree". Browser will redirect to `http://127.0.0.1:3000/callback?code=LONG_CODE&state=xyz` (the server doesn't need to run — just copy the `code` value from the URL).

**Step 3 — Exchange code for refresh_token:**
Run this curl command (replace placeholders):
```bash
curl -X POST https://accounts.spotify.com/api/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -H "Authorization: Basic $(echo -n 'YOUR_CLIENT_ID:YOUR_CLIENT_SECRET' | base64)" \
  -d "grant_type=authorization_code&code=YOUR_CODE&redirect_uri=http%3A%2F%2F127.0.0.1%3A3000%2Fcallback"
```
Response JSON contains `refresh_token`. Copy it.

**Step 4 — Add GitHub Secrets:**
Go to GitHub repo → Settings → Secrets and variables → Actions → New repository secret:
- `SPOTIFY_CLIENT_ID` — from Step 1
- `SPOTIFY_CLIENT_SECRET` — from Step 1
- `SPOTIFY_REFRESH_TOKEN` — from Step 3 response

**Step 5 — Verify:**
After adding secrets, manually trigger the refresh-data.yml workflow (GitHub Actions → Refresh Data → Run workflow) and check the run logs. The "Fetch Spotify data" step should complete without errors and the commit should show updated spotify data in data.json.

**Refresh Token Rotation Warning:**
Spotify may occasionally return a new refresh_token in the token refresh response. If the pipeline log shows `Spotify returned a new refresh token — update SPOTIFY_REFRESH_TOKEN secret manually: <token>`, update the `SPOTIFY_REFRESH_TOKEN` GitHub Secret with the new value. Ignoring this will eventually cause pipeline failures.
